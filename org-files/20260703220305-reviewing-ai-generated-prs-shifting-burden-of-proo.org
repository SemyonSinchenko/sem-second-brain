* Summary
Source: [[https://gerdzellweger.com/engineering/2026/06/27/prs-and-llms.html][https://gerdzellweger.com/engineering/2026/06/27/prs-and-llms.html]]
Senior engineers are increasingly overwhelmed by the volume of Pull Requests generated by coding agents. The core challenge is shifting the burden of proof from the reviewer to the author (and the agent). Strategies include enforcing high test coverage, autonomous end-to-end testing by agents, and developing executable specifications for APIs to validate AI-generated code against strict oracles.

* Key Takeaways
- Senior engineers report reviewing AI-generated PRs consumes time previously spent writing code.
- Trust issues arise because AI code often lacks algorithmic efficiency, clean structure, or adherence to API surfaces.
- Merging changes faster than the codebase can absorb creates risk, especially in complex or underspecified modules.
- Solution strategy: Shift burden of proof to the PR author/agent via rigorous automated testing and executable specs.
- Evaluation criteria for AI code: Functionality, complexity, API surface, clean code principles, and performance.

* Notes
** The Three Core Complaints
1. *Reviewing consumes writing time:* Humans spend most of their day reviewing agent output rather than coding.
2. *Quality perception:* AI-generated code is often not perceived as 'good enough' by experienced engineers.
3. *Absorption rate:* Large changes are merged faster than the team can understand or maintain them.

** Feldera's Strategy: Shifting the Burden of Proof
Instead of removing human reviews, the requirement for PR readiness is moved to the author/agent.
- *Coverage Targets:* Set 99% coverage targets for new repositories. LLMs can write missing tests once gaps are identified by tools like =llvm-cov=.
- *Test Validation:* New tests must be validated to fail by reverting the change, ensuring they capture real issues.
- *Autonomous E2E Testing:* Each substantial PR must trigger an agent to run the product end-to-end across different configurations and failure cases.
- *Executable Specifications:* APIs used by clients must include an executable specification or oracle. Property-based testing runs generated inputs through both the spec and the implementation.
- *Interchangeable Specs:* Design specs and implementations to be interchangeable to isolate bugs easily.

** Evaluating AI Code Quality
To make reviews less subjective, code is evaluated along five points:
1. It works (requires clear expected behavior and test corpus).
2. It uses the right data structures and algorithmic complexity.
3. It exposes and uses the right API surface.
4. It follows clean code principles (DRY, etc.).
5. It is fast enough.

** Codebase Health and Agents
Institutional memory regarding "risky" code areas is vital. Agents perform worse than humans in modules riddled with footguns.
- AI changes the cost of leaving code in a bad state; it is now more viable to refactor "god modules" or underspecified code.
